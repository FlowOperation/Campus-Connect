# Documentation Guidelines

## Principles
- Documents must drive implementation decisions.
- Keep docs versioned with code changes.
- Prefer concrete examples over generic statements.

## Required Docs per Significant Feature
- Feature specification
- Data model changes
- Security impact notes
- Test plan and rollout notes
- ADR entry for architecture-level decisions

## Style Guide
- Use concise headings and structured sections.
- Include explicit assumptions and unresolved questions.
- Add links to source files and related docs.

## Review Rules
- Docs are part of code review for architecture/security changes.
- Outdated docs should be corrected in same PR when feasible.
